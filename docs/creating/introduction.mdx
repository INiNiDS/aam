# Introduction to AAM Development

This category helps you create your own plugin or library for AAM in your preferred programming language.

For example, if you need AAM for Rust, you might use the [official `aam-rs` library](https://crates.io/crates/aaml). If there is no library for your language (e.g., Python, Go, TypeScript), you can create one from scratch! The `aam-rs` implementation is considered the **gold standard** for AAM library design and serves as the prime reference for how a robust AAM library should be structured.

## Getting Started

First, check if a library already exists for your language.
* **If yes:** Contribute! Add new features, fix bugs, or improve documentation.
* **If no:** Create one from scratch following the guidelines in this section.

## The Configuration Engine Paradigm

If you are building a library, you need to understand what AAM actually is. **AAM is a declarative text configuration format** built on simple key-value pairs, but it excels through powerful structural capabilities like file imports, aliases, recursive deep resolution, and circular dependency safety.

Historically, AAM libraries were conceptualized as basic sets of "parsers, serializers, and validators." However, modern AAM libraries—modeled after `aam-rs`—act as comprehensive **Configuration Engines**. A mature, high-quality AAM library provides the following three core mechanics:

1. **Parser & Loader Engine:** Reading raw text (`.aam` files) or strings, and seamlessly handling inline `@import` directives by traversing and merging external files into a unified in-memory data structure (analogous to a Python `dict`, a Go `map`, or a Rust `struct`).
2. **Smart Query Engine:** Allowing users to safely and efficiently traverse the configuration graph. This means providing bidirectional lookups (finding keys by their value, or values by their key) and recursive "deep" aliasing resolution with built-in loop detection to prevent stack overflows.
3. **Configuration Builder:** Giving developers a programmatic, fluent API to iteratively construct, manipulate, and save `.aam` configurations dynamically via code.

When you author a new library, your goal is to translate these powerful mechanics into idiomatic code that feels entirely natural to your target language's ecosystem.
