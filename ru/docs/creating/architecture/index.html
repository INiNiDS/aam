Архитектура библиотеки

Каждая современная AAM-библиотека, независимо от целевого языка, строится как конфигурационный движок. По образцу эталонной реализации aam-rs движок состоит из трёх основных подсистем. Понимание их ответственности — первый шаг к корректной реализации.

┌─────────────┐     текст     ┌─────────────┐   Объект AAML   ┌─────────────┐
│  файл .aam  │  ──────────►  │  Загрузчик  │  ─────────────► │ Движок запр.│
└─────────────┘               └──────┬──────┘                 └─────────────┘
                                     │  ▲
                               Объект AAML (слитый)
                                     │  │
                               ┌─────▼───────┐
                               │ Конструктор │
                               └─────────────┘
                                     │
                                     │  текст
                                     ▼
                               ┌─────────────┐
                               │  файл .aam  │
                               └─────────────┘

1. Парсер и загрузчик​

Загрузчик — точка входа вашей библиотеки. Он читает необработанный AAM-текст (или путь к файлу) и создаёт унифицированное представление в памяти — обычно называемое AAML или эквивалентной структурой/картой на вашем языке.

Ответственность​

  1. Токенизация и разбор — Разбивка исходного текста на пары ключ—значение, строки и директивы (@import, @schema, @type).
  2. Загрузка и слияние файлов — При обнаружении директивы @import "other.aam" загрузчик читает этот файл, разбирает его и сливает содержимое с текущим документом.
  3. Построение модели данных — Создание внутренней карты, хранящей граф конфигурации.

Внутренняя модель данных​

Минимальный объект AAML в памяти обычно выглядит примерно так (на Rust, Python и Go):

Rust (struct)

pub struct AAML {
    /// Сохраняет порядок вставки для детерминированной сериализации.
    pub elements: IndexMap<String, String>,
    pub schemas: HashMap<String, Schema>,
}

Python (dict)

class AAML:
    def __init__(self):
        # Словари Python сохраняют порядок вставки в современных версиях
        self.elements: dict[str, str] = {}
        self.schemas: dict[str, Schema] = {}

Go (map и срезы)

type AAML struct {
    // Карты Go не сохраняют порядок, поэтому часто используется отдельный срез ключей
    Elements map[string]string
    Keys     []string
    Schemas  map[string]Schema
}

2. Движок запросов​

Движок запросов делает AAM мощным инструментом. Вместо прямого доступа к карте данных библиотека предоставляет умные методы обхода для чтения конфигурации.

Ответственность​

  1. Двунаправленный поиск (find_obj) — Иногда разработчик знает ключ, иногда — значение и хочет найти ключ. Движок должен сначала пробовать прямой поиск по ключу, а при неудаче — обратный поиск.
  2. Рекурсивное разрешение (find_deep) — Пользователи AAM часто используют псевдонимы (например, current_path = base_path). Движок запросов должен следовать цепочке значений до терминального значения, которое не является ключом.
  3. Обнаружение циклов — Поскольку AAM декларативен, пользователи могут легко создавать циклические ссылки (например, a = b и b = a). Функция find_deep обязана обнаруживать циклы и возвращать последнее уникальное посещённое значение, предотвращая бесконечную рекурсию и переполнение стека.
  4. Обёртка абстрактных значений (FoundValue) — Вместо возврата сырых строк оборачивайте результат в насыщенный тип (класс или структуру), позволяющий удобно приводить значение к целым числам, булевым значениям или выполнять строковые операции.

Поток запросов​

При вызове config.find_deep(key):

Фаза 1: Прямой поиск
  └── Попытка config[key] -> найденное значение V

Фаза 2: Рекурсия
  └── Обнаружение цикла: встречали ли мы V? Если да — прервать и вернуть.
  └── Является ли V ключом в config?
      ├── Если да: V становится новым ключом, переход к Фазе 1.
      └── Если нет: вернуть V как окончательное разрешённое значение.

3. Конструктор конфигураций​

Конструктор выполняет обратный поток применения: он позволяет программно генерировать AAM-текст безопасно и эффективно, без ручной конкатенации строк.

Ответственность​

  1. Генерация гибкого API — Предоставление методов для последовательного добавления пар ключ—значение, комментариев, псевдонимов типов и блоков @schema.
  2. Детерминированный вывод — Конструктор должен сохранять точный порядок вставки элементов, чтобы генерируемые файлы .aam оставались чистыми и удобными для сравнения.
  3. Валидация и форматирование — Строки с пробелами должны автоматически заключаться в кавычки, переносы строк — расставляться корректно, а комментарии — начинаться с #.

Минимальный контракт вывода​

Для программной последовательности построения (например, на TypeScript):

const builder = new AAMBuilder();
builder.comment("Конфигурация сети");
builder.addLine("host", "localhost");
builder.addLine("port", "8080");

Конструктор должен безупречно сериализовать это в:

# Конфигурация сети
host = localhost
port = 8080

Всё вместе​

Типичное взаимодействие с библиотекой по модели конфигурационного движка (на Python):

import aam

# 1. Разбор и загрузка (все @import обрабатываются автоматически)
config = aam.load("config.aam")

# 2. Умный запрос (глубокое разрешение)
terminal_path = config.find_deep("app_path")
print(f"Разрешённый путь: {terminal_path}")

# 3. Программное создание новой конфигурации
builder = aam.AAMBuilder()
builder.comment("Сгенерированные переопределения")
