# Проверка типов

На этапе разбора высококачественный движок загрузки должен подтвердить, что **каждое значение совместимо с объявленным типом**. Эта страница описывает, как выполнить такую проверку для каждой категории типов AAM перед финализацией карты `AAML`.

---

## Обзор

Проверка типов выполняется в два этапа:

1. **Разрешение** — разворачивание псевдонимов `@type` до их канонического типа (см. [`@type`](../directives/type)).
2. **Проверка** — сравнение разобранного узла времени выполнения с каноническим типом.

```rust
fn check_value(
    key: &str,
    value: &ParsedValue, // Внутренний узел до сохранения в AAML
    declared_ty: &String,
    registry: &HashMap<String, String>,
) -> Result<(), AamlError> {
    let canonical = resolve_type(declared_ty, registry)?;
    check_value_canonical(key, value, &canonical)
}
```

## Примитивные типы

Примитивные типы — простейший случай: проверяем, что вариант `PrimitiveTypes` соответствует ожидаемому тэгу типа:

| Тип AAM | Вариант `AamValue` | Примеры допустимых литералов |
|---------|---------------------|------------------------------|
| `i32` | `PrimitiveTypes::I32` | `42`, `-7`, `0` |
| `f64` | `PrimitiveTypes::F64` | `3.14`, `-0.5`, `1.0` |
| `string` | `PrimitiveTypes::Str` | `"hello"`, `""` |
| `bool` | `PrimitiveTypes::Bool` | `true`, `false` |
| `color` | `PrimitiveTypes::Color` | `#ff0000`, `#0a0b0c` |

```rust
fn check_value_canonical(
    key: &str,
    value: &ParsedValue,
    ty: &String, // Или внутреннее перечисление, представляющее примитивный тип
) -> Result<(), AamlError> {
    let ok = match (ty.as_str(), value) {
        ("i32",    ParsedValue::I32(_))   => true,
        ("f64",    ParsedValue::F64(_))   => true,
        ("string", ParsedValue::Str(_))   => true,
        ("bool",   ParsedValue::Bool(_))  => true,
        ("color",  ParsedValue::Color(_)) => true,
        _ => false,
    };

    if !ok {
        return Err(AamlError::TypeMismatch {
            key: key.to_string(),
            expected: ty.to_string(),
            found: value.type_name(),
        });
    }
    Ok(())
}
```

:::tip Числовое приведение
AAM **не** выполняет неявное приведение `i32` к `f64` или наоборот. `42` всегда является `i32`; `42.0` всегда является `f64`. Не добавляйте неявные приведения.
:::

## Комплексные типы с пространством имён

Комплексные типы находятся в трёх пространствах имён: `types::physics::`, `types::math::` и `types::time::`. На уровне текста AAM они все хранятся как `AamValue::Complex`, который оборачивает скалярное значение вместе с тэгом единицы измерения.

### Представление

```rust
pub struct ComplexValue {
    /// Скалярная величина, всегда хранится как f64 внутренне.
    pub magnitude: f64,
    /// Полностью квалифицированный ключ типа, например "physics::kilogram".
    pub type_key: String,
}

pub enum AamValue {
    // ...примитивы...
    Complex(ComplexValue),
    MathValue(MathValue),   // векторы, матрицы, кватернионы
    TimeValue(TimeValue),   // datetime, duration, единицы времени
}
```

### Физические типы (`types::physics::*`)

Все физические типы представляют скалярную величину с единицей СИ. В нативных типах вашего языка отображайте величину в `f64` (или эквивалент) и прикрепляйте единицу как метаданные:

| Тип AAM | Рекомендуемое нативное отображение |
|---------|-------------------------------------|
| `types::physics::kilogram` | `f64` (масса в кг) |
| `types::physics::meter` | `f64` (длина в м) |
| `types::physics::meterpersecond` | `f64` (скорость в м/с) |
| `types::physics::newton` | `f64` (сила в Н) |
| `types::physics::kelvin` | `f64` (температура в К) |
| … | `f64` + тэг единицы |

Проверка: поле `type_key` в `ComplexValue` должно точно совпадать с ожидаемым ключом физического типа.

```rust
(AamType::Physics(expected_unit), AamValue::Complex(v)) => {
    v.type_key == expected_unit.key()
}
```

### Математические типы (`types::math::*`)

Математические значения — многокомпонентные структуры. Каждый компонент является `f64`:

| Тип AAM | Нативное отображение |
|---------|----------------------|
| `types::math::vector2` | `{ x: f64, y: f64 }` |
| `types::math::vector3` | `{ x: f64, y: f64, z: f64 }` |
| `types::math::vector4` | `{ x: f64, y: f64, z: f64, w: f64 }` |
| `types::math::quaternion` | `{ x: f64, y: f64, z: f64, w: f64 }` |
| `types::math::matrix3x3` | `[[f64; 3]; 3]` |
| `types::math::matrix4x4` | `[[f64; 4]; 4]` |

Проверка: убедитесь, что вариант `AamValue::MathValue` соответствует ожидаемому подтипу math и количество компонентов корректно.

### Временны́е типы (`types::time::*`)

Временны́е значения используют **ISO 8601** в качестве текстового представления. Отображайте их на ближайший нативный тип целевого языка:

| Тип AAM | Текстовый формат | Рекомендуемое нативное отображение |
|---------|-----------------|-------------------------------------|
| `types::time::datetime` | `"2024-06-01T12:00:00Z"` | `DateTime` / `chrono::DateTime` |
| `types::time::duration` | `"PT1H30M"` | `Duration` / `chrono::Duration` |
| `types::time::year` | `2024` | `u32` |
| `types::time::day` | `1`–`365` | `u16` |
| `types::time::hour` | `0`–`23` | `u8` |
| `types::time::minute` | `0`–`59` | `u8` |

```rust
// Пример: валидация time::datetime
(AamType::Time(TimeType::DateTime), AamValue::TimeValue(TimeValue::DateTime(dt))) => {
    // dt уже разобранный DateTime — дополнительная проверка не нужна
    true
}
(AamType::Time(TimeType::DateTime), AamValue::Str(s)) => {
    // Fallback: разбираем строку как ISO 8601 и сохраняем результат
    DateTime::parse_from_rfc3339(s).is_ok()
}
```

:::note
Если в целевом языке нет нативного типа `DateTime` (например, в чистом C), сохраняйте строку ISO 8601 напрямую и предоставляйте вспомогательную функцию для её разбора. Чётко документируйте это ограничение в README вашей библиотеки.
:::

## Типы списков

Значение `list<T>` допустимо тогда и только тогда, когда **каждый элемент** удовлетворяет типу `T`:

```rust
(AamType::List(inner_ty), AamValue::List(items)) => {
    for (i, item) in items.iter().enumerate() {
        check_value_canonical(
            &format!("{key}[{i}]"),
            item,
            inner_ty,
        )?;
    }
    Ok(())
}
```

Вложенные списки (`list<list<i32>>`) должны обрабатываться рекурсивно.

## Типы схем

Когда тип поля является именем `@schema`, валидируйте значение как **вложенный объект** по этой схеме:

```rust
(AamlType::Schema(schema_name), ParsedValue::Object(obj)) => {
    let schema = registry
        .schemas
        .get(schema_name)
        .ok_or_else(|| AamlError::UnknownSchema { name: schema_name.clone() })?;
    validate_against_schema(obj, schema, registry)?;
    Ok(())
}
```

## Вывод ошибок

Все ошибки типов должны включать:

- **Ключ** (или путь поля, например `players[0].name`), на котором произошёл сбой.
- **Ожидаемый тип** в виде строки для человека.
- **Фактическое значение**, которое было найдено.
- **Расположение в источнике** (строка, столбец), если ваш парсер отслеживает позиции.

```
Error: type mismatch
  key:      `spawn_point`
  expected: math::vector3
  found:    string ("0.0, 1.0, 2.0")
  --> level.aam:12:14

hint: use the structured form: spawn_point = {x: 0.0, y: 1.0, z: 2.0}
```

Собирайте **все** ошибки перед возвратом — не прерывайте работу на первом несоответствии. Это позволяет пользователю исправить все проблемы за один цикл редактирования.

## Контрольный список валидации

| Проверка | Обязательна |
|----------|-------------|
| Примитивное значение соответствует объявленному примитивному типу | ✅ |
| `type_key` комплексного значения соответствует ожидаемой единице | ✅ |
| `time::datetime` разбирается как допустимый ISO 8601 | ✅ |
| Каждый элемент `list<T>` соответствует `T` | ✅ |
| Псевдонимы `@type` разрешены до сравнения | ✅ |
| Обязательные поля схемы присутствуют | ✅ |
| Нет неизвестных полей схемы (строгий режим) | ✅ |
| Вложенные объекты схем валидируются рекурсивно | ✅ |
| Нет неявного числового приведения (`i32` ↔ `f64`) | ✅ |
