# Стандарты API

На этой странице определены **межязыковые стандарты**, которым обязана следовать каждая AAM-библиотека. Соблюдение этих стандартов гарантирует, что файлы AAM, написанные для одной библиотеки, корректно работают с другой, а разработчики, переходящие между языками, встречают знакомый API. В качестве базы мы используем эталонный движок (`aam-rs`).

---

## Стандартные методы API

Для обеспечения единообразного опыта «конфигурационного движка» библиотеки должны реализовывать следующие основные методы (или их ближайшие идиоматические эквиваленты в целевом языке).

### 1. Основной API загрузчика

Эти методы управляют разбором и слиянием необработанного текста в структуру `AAML`:

| Общий метод | Описание |
|---|---|
| `parse(content: String)` | Разбирает сырую строку в объект в памяти. |
| `load(file_path: Path)` | Загружает и разбирает файл с диска, автоматически разрешая директивы `@import`. |
| `merge_content(content: String)` | Разбирает строку и сливает её с *текущим* объектом, перезаписывая существующие ключи. |
| `merge_file(file_path: Path)` | Загружает файл и сливает его с текущим объектом. |

### 2. API запросов

Эти методы определяют, как разработчики читают данные из дерева:

| Общий метод | Описание |
|---|---|
| `find_obj(key: String)` | Умный двунаправленный поиск. Если `key` существует — возвращает его значение. Если нет — возвращает ключ, значением которого является `key`. |
| `find_deep(key: String)` | Рекурсивный поиск. Следует по цепочке значений до терминального значения, которое не является ключом. Обязательно должна быть защита от циклов! |
| `find_key(value: String)` | Строгий обратный поиск. Находит первый ключ, содержащий указанное значение. |

### 3. API конструктора

Конструкт `AAMBuilder` должен предоставлять:

| Общий метод | Описание |
|---|---|
| `add_line(key, value)` | Добавляет стандартную пару `key = value`. |
| `comment(text)` | Добавляет строку, начинающуюся с `# `. |
| `schema(name, fields[])` | Генерирует встроенную директиву `@schema`. |
| `schema_multiline(...)` | Генерирует многострочную директиву `@schema`. |
| `derive(path, schemas[])` | Генерирует директиву `@derive`. |
| `type_alias(alias, type)` | Генерирует директиву `@type`. |
| `to_file(path)` / `to_string()` | Записывает детерминированный конструктор на диск или в память. |

## Версионирование

Библиотеки должны следовать [Семантическому версионированию](https://semver.org/):

- **Патч-выпуск** (`1.0.x`) может исправлять ошибки без изменения поведения.
- **Минорный выпуск** (`1.x.0`) может добавлять новые необязательные возможности.
- **Мажорный выпуск** (`x.0.0`) может вводить критические изменения API.

В `README` библиотеки должна быть чётко указана **версия спецификации AAM**, которую она реализует (например, `AAM 1.0`).

## Обработка ошибок

Все публичные методы, которые могут завершиться с ошибкой, должны возвращать структурированный тип результата — **никогда** не паниковать и не бросать нетипизированное исключение при ошибке пользователя:

| Язык | Ожидаемый паттерн |
|-------------|----------------------------------------------|
| Rust        | `Result<T, AamlError>`                       |
| Go          | `(T, error)`                                 |
| Python      | raise `AamlError` (подкласс `Exception`)     |
| TypeScript  | throw `AamlError` (подкласс `Error`)         |
| Java/Kotlin | throw `AamlException` (checked или документированный)|

Варианты ошибок должны быть задокументированы. Как минимум различайте:

- **Ошибки ввода/вывода** (файл не найден, доступ запрещён)
- **Ошибки разбора** (неверный синтаксис), включая номера строки и столбца
- **Ограничения поиска** (ключ не найден)

## Детерминированный вывод

Вывод сериализации должен быть **детерминированным** для данного ввода. Двойной вызов `builder.to_string()` для одного документа в памяти должен давать побайтово идентичный текст. Это позволяет инструментам надёжно сравнивать файлы `.aam`.

---

## Пробелы и форматирование

При сериализации конструкторы должны создавать вывод, соответствующий правилам форматирования AAM:

- Ровно **один пробел** с каждой стороны `=`.
- Директивы (`@import`, `@schema` и т. д.) располагаются **перед** парами ключ—значение.
- Тела встроенных схем разделяются запятыми; многострочные тела — переносами строк.
- Никакого завершающего пробела на любой строке.
- Файл заканчивается ровно **одним символом переноса строки**.

---

## Кодировка

Все файлы `.aam` кодируются в **UTF-8**. Библиотеки должны:

- Читать файлы как UTF-8 и возвращать чёткую ошибку, если файл не является допустимым UTF-8.
- Записывать файлы как UTF-8 без BOM.

---

## Потокобезопасность

Разобранные объекты `AAML` должны быть **потокобезопасными для чтения**. Если язык поддерживает это, пометьте тип как `Send + Sync` (Rust) или аналог, чтобы документы можно было разделять между потоками без блокировок.

Если ваша библиотека предоставляет изменяемый конструктор (`AAMBuilder`), он **по умолчанию не обязан** быть потокобезопасным, но это должно быть чётко задокументировано.

---

## Тестовый набор

Каждая библиотека должна поставляться с тестовым набором, охватывающим:

- [ ] Базовый разбор `key = value` для всех примитивных типов
- [ ] Разрешение файлов `@import` и объединение
- [ ] Корректность форматирования `@schema`
- [ ] `find_obj` корректно определяет обратные ключи
- [ ] `find_deep` предотвращает бесконечные циклы при зависимости `A -> A`
- [ ] Полный цикл сериализатора: `parse(builder.to_string()) == parsed_state`
