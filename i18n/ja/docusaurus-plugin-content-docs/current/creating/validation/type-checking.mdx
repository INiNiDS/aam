# 型チェック

パースフェーズ中に、高品質なローダーエンジンは**すべての値が宣言された型と互換性があること**を確認する必要があります。このページでは、`AAML` マップを確定する前にAAM型の各カテゴリに対してそのチェックを実行する方法を説明します。

---

## 概要

型チェックは2つのフェーズで進行します：

1. **解決** — `@type` エイリアスを正規型に展開します（[`@type`](../directives/type) 参照）。
2. **チェック** — パースされたランタイムノードを正規型と比較します。

---

## プリミティブ型

プリミティブ型が最も単純なケースです — `PrimitiveTypes` バリアントが期待される型タグと一致することを確認します：

| AAM型 | `AamValue` バリアント | 受け入れられるリテラル例 |
|-------|-----------------------|----------------------|
| `i32` | `PrimitiveTypes::I32` | `42`, `-7`, `0` |
| `f64` | `PrimitiveTypes::F64` | `3.14`, `-0.5`, `1.0` |
| `string` | `PrimitiveTypes::Str` | `"hello"`, `""` |
| `bool` | `PrimitiveTypes::Bool` | `true`, `false` |
| `color` | `PrimitiveTypes::Color` | `#ff0000`, `#0a0b0c` |

:::tip 数値の型強制
AAMは `i32` を `f64` に、またはその逆に**暗黙的に型強制しません**。`42` は常に `i32`；`42.0` は常に `f64` です。暗黙的な型強制を追加しないでください。
:::

---

## バリデーションチェックリスト

| チェック | 必須 |
|---------|------|
| プリミティブ値が宣言されたプリミティブ型と一致する | ✅ |
| 複合値の `type_key` が期待されるユニットと一致する | ✅ |
| `time::datetime` が有効なISO 8601としてパースされる | ✅ |
| `list<T>` のすべての要素が `T` と一致する | ✅ |
| `@type` エイリアスが比較前に解決される | ✅ |
| 必須スキーマフィールドが存在する | ✅ |
| 不明なスキーマフィールドがない（厳格モード） | ✅ |
| ネストされたスキーマオブジェクトが再帰的に検証される | ✅ |
| 暗黙的な数値型強制がない（`i32` ↔ `f64`） | ✅ |
