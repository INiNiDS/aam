ライブラリアーキテクチャ

対象言語に関係なく、すべての現代的なAAMライブラリは設定エンジンとして構築されます。参照実装 aam-rs のアーキテクチャに従い、エンジンは3つの主要なサブシステムで構成されます。各サブシステムの責務を理解することが、正しい実装への第一歩です。

┌─────────────┐     テキスト    ┌─────────────┐   AAMLオブジェクト  ┌─────────────┐
│  .aamファイル │  ──────────►  │   ローダー   │  ─────────────► │クエリエンジン│
└─────────────┘               └──────┬──────┘                 └─────────────┘
                                     │  ▲
                               AAMLオブジェクト（マージ済み）
                                     │  │
                               ┌─────▼───────┐
                               │   ビルダー   │
                               └─────────────┘
                                     │
                                     │  テキスト
                                     ▼
                               ┌─────────────┐
                               │  .aamファイル │
                               └─────────────┘

1. パーサー＆ローダー​

ローダーはライブラリのエントリーポイントです。生のAAMテキスト（またはファイルパス）を読み取り、統合されたインメモリ表現（通常 AAML または対象言語における同等の構造体/マップ）を生成します。

責務​

  1. トークン化とパース — ソーステキストをキーと値のペア、文字列、およびディレクティブ（@import、@schema、@type）に分割します。
  2. ファイルの読み込みとマージ — @import "other.aam" ディレクティブが検出された場合、ローダーはそのファイルをシームレスに読み取り、パースして内容を現在のドキュメントにマージします。
  3. データモデルの構築 — 設定グラフを格納する内部マップを構築します。

内部データモデル​

メモリ上の最小限の AAML オブジェクトは通常次のようになります（Rust、Python、Goで示します）：

Rust（struct）

pub struct AAML {
    /// 決定論的なシリアライゼーションのために挿入順序を保持します。
    pub elements: IndexMap<String, String>,
    pub schemas: HashMap<String, Schema>,
}

Python（dict）

class AAML:
    def __init__(self):
        # Pythonの辞書は最新バージョンで挿入順序を保持します
        self.elements: dict[str, str] = {}
        self.schemas: dict[str, Schema] = {}

Go（map とスライス）

type AAML struct {
    // Goのマップは順序を保持しないため、別のキースライスがよく使われます
    Elements map[string]string
    Keys     []string
    Schemas  map[string]Schema
}

2. クエリエンジン​

クエリエンジンがAAMを強力にします。生のマップを直接公開する代わりに、ライブラリは設定を読み取るためのスマートなトラバーサルメソッドを提供します。

責務​

  1. 双方向ルックアップ（find_obj） — 開発者がキーを知っている場合と、値を知ってキーを見つけたい場合があります。エンジンはまず直接キールックアップを試み、失敗した場合は逆ルックアップを実行します。
  2. 再帰的解決（find_deep） — AAMユーザーはエイリアスをよく使用します（例：current_path = base_path）。クエリエンジンはキーでない終端値を見つけるまで値の連鎖をたどる必要があります。
  3. ループ検出 — AAMは宣言型であるため、ユーザーは循環参照を簡単に作成する可能性があります（例：a = b と b = a）。find_deep 関数はサイクルを検出し、無限再帰とスタックオーバーフローを防ぐために最後に訪問した一意の値を返す必要があります。
  4. 抽象的な値ラッピング（FoundValue） — 生の文字列を返す代わりに、結果を整数、ブール値へのキャストや、その場での文字列操作を簡単にするリッチな型（カスタムクラスまたは構造体）にラップします。

クエリエンジンのフロー​

ユーザーが config.find_deep(key) を呼び出したとき：

フェーズ1：直接ルックアップ
  └── config[key] を試みる -> 値V を取得

フェーズ2：再帰
  └── ループ検出：Vはすでに見たか？ Yesなら中断して返す。
  └── VはconfigのキーとしてExistしているか？
      ├── Yesなら、VはフェーズI1の新しいキーになる。
      └── Noなら、Vを最終解決値として返す。

3. 設定ビルダー​

ビルダーは逆のアプリケーションフローを実行します：文字列を手動で連結することなく、ユーザーがAAMテキストをプログラム的に安全かつ効率的に生成できます。

責務​

  1. 流暢なAPI生成 — キーと値のペア、コメント、型エイリアス、@schema ブロックを順次追加するメソッドを提供します。
  2. 決定論的な出力 — ビルダーは生成された .aam ファイルがクリーンで差分可能なままであるように要素の正確な挿入順序を保持する必要があります。
  3. バリデーションとフォーマット — スペースを含む文字列は自動的に引用符で囲まれ、改行は正しく配置され、コメントは # でプレフィックスされます。

最小出力コントラクト​

プログラム的なビルドシーケンス（例：TypeScript）が与えられた場合：

const builder = new AAMBuilder();
builder.comment("ネットワーク設定");
builder.addLine("host", "localhost");
builder.addLine("port", "8080");

ビルダーは完璧にシリアライズします：

# ネットワーク設定
host = localhost
port = 8080

すべてを統合する​

設定エンジンモデルに従った典型的なライブラリの操作（Pythonで示します）：

import aam

# 1. パースとロード（すべての@importを自動的に処理）
config = aam.load("config.aam")

# 2. スマートクエリ（深い解決）
terminal_path = config.find_deep("app_path")
print(f"解決されたパス: {terminal_path}")

# 3. プログラム的に新しい設定を作成
builder = aam.AAMBuilder()
builder.comment("生成されたオーバーライド")
builder.add_line("app_path", terminal_path)
builder.to_file("overrides.aam")

これら3つのコアシステムを分離することで、ライブラリは開発者に完全な制御を提供します：任意の設定を解析し、複雑な依存関係ウェブを安全に解決し、または堅牢なファイルをプログラム的に生成できます。
