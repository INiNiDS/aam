API標準

このページは、すべてのAAMライブラリが準拠すべきクロス言語標準を定義します。これらの標準に従うことで、あるライブラリ用に書かれたAAMファイルが別のライブラリでも正しく動作し、言語を切り替える開発者が親しみやすいAPIに出会えます。参照エンジン（aam-rs）をベースラインとして使用します。

標準APIメソッド​

一貫した「設定エンジン」体験を提供するために、ライブラリは以下のコアメソッド（または対象言語における最も近いイディオマティックな同等物）を実装する必要があります。

1. コアローダーAPI​

これらのメソッドは生のテキストを AAML 構造体にパースしてマージすることを管理します：

汎用メソッド説明
parse(content: String)生の文字列をインメモリオブジェクトにパースします。
load(file_path: Path)ディスクからファイルを読み込み、@import ディレクティブを自動的に解決します。
merge_content(content: String)文字列をパースして既存の 現在の オブジェクトにマージし、既存のキーを上書きします。
merge_file(file_path: Path)ファイルを読み込んで現在のオブジェクトにマージします。

2. クエリAPI​

これらのメソッドは開発者がツリーからどのように読み取るかを制御します：

汎用メソッド説明
find_obj(key: String)スマートな双方向ルックアップ。key が存在すれば値を返します。存在しない場合、値が key のキーを返します。
find_deep(key: String)再帰的検索。キーでない終端値に達するまで値の連鎖をたどります。ループ検出が必须です！
find_key(value: String)厳格な逆引きルックアップ。指定した値を保持する最初のキーを見つけます。

3. ビルダーAPI​

AAMBuilder コンストラクトは以下を提供する必要があります：

汎用メソッド説明
add_line(key, value)標準的な key = value ペアを追加します。
comment(text)# で始まる行を追加します。
schema(name, fields[])インライン @schema ディレクティブを生成します。
schema_multiline(...)複数行の @schema ディレクティブを生成します。
derive(path, schemas[])@derive ディレクティブを生成します。
type_alias(alias, type)@type ディレクティブを生成します。
to_file(path) / to_string()決定論的なビルダーをディスクまたはメモリにフラッシュします。

バージョニング​

ライブラリはセマンティックバージョニングに従うべきです：

  • パッチリリース（1.0.x）は動作を変更せずにバグを修正できます。
  • マイナーリリース（1.x.0）は新しいオプション機能を追加できます。
  • メジャーリリース（x.0.0）は破壊的なAPI変更を導入できます。

ライブラリの README は、実装する AAM仕様バージョン（例：AAM 1.0）を明確に記載する必要があります。

エラーハンドリング​

失敗する可能性のあるすべてのパブリックメソッドは構造化された結果型を返す必要があり、ユーザーエラーでパニックしたり型なしの例外をスローしてはなりません：

言語期待されるパターン
RustResult<T, AamlError>
Go(T, error)
PythonAamlError（Exception のサブクラス）をレイズ
TypeScriptAamlError（Error のサブクラス）をスロー
Java/KotlinAamlException（チェック済みまたはドキュメント化済み）をスロー

エラーバリアントはドキュメント化される必要があります。最低限、以下を区別してください：

  • I/Oエラー（ファイルが見つからない、アクセス拒否）
  • パースエラー（無効な構文）、行番号と列番号を含む
  • ルックアップ制約（キーが見つからない）

決定論的な出力​

シリアライズの出力は特定の入力に対して決定論的でなければなりません。同じインメモリドキュメントで builder.to_string() を2回実行すると、バイト単位で同一のテキストが生成される必要があります。これにより、ツールが .aam ファイルを確実に差分できます。

空白とフォーマット​

シリアライズ時、ビルダーはAAMフォーマットルールに準拠した出力を生成する必要があります：

  • = の両側にそれぞれスペース1つ。
  • ディレクティブ（@import、@schema など）はキーと値のエントリの前に配置します。
  • インラインスキーマ本体はカンマを使用；複数行本体は改行を使用。
  • 行末に空白なし。
  • ファイルは正確に改行文字1つで終わります。

エンコーディング​

すべての .aam ファイルは UTF-8 でエンコードされています。ライブラリは：

  • ファイルをUTF-8として読み、ファイルが有効なUTF-8でない場合は明確なエラーを返す必要があります。
  • BOMなしでUTF-8としてファイルを書き込む必要があります。

スレッドセーフティ​

パースされた AAML オブジェクトは読み取りに対してスレッドセーフであるべきです。言語がサポートする場合、型を Send + Sync（Rust）または同等のものとしてマークして、ロックなしでスレッド間でドキュメントを共有できるようにしてください。

ライブラリが変更可能なビルダー（AAMBuilder）を公開する場合、デフォルトでスレッドセーフである必要はありませんが、これを明確にドキュメント化してください。

テストスイート​

すべてのライブラリは以下をカバーするテストスイートを同梱する必要があります：

  • すべてのプリミティブ型の基本的な key = value パース
  • @import ファイル解決と追加
  • @schema フォーマットの正確性
  • find_obj が逆キーを正しく識別する
  • find_deep が A -> A 依存関係でのインフィニットループを防ぐ
  • シリアライザーのラウンドトリップ：parse(builder.to_string()) == parsed_state
