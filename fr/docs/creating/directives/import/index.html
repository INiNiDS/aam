La Directive @import

La directive @import instruit le parser à fusionner un fichier .aam externe dans le document actuel. Toutes les paires clé-valeur, les alias de type et les définitions de schéma du fichier importado deviennent disponibles dans le scope actuel.

@import chemin/vers/autre.aam

Syntaxe​

@import prend une unique chaîne citée — le chemin vers le fichier à importer :

@import "shared/defaults.aam"
@import "../common/colors.aam"

Les chemins sont résolus relativement au fichier qui contient la directive, non au répertoire de travail du processus.

Ce que fait @import​

Lorsque le parser rencontre @import "fichier.aam" :

  1. Résout le chemin relativement au répertoire du fichier actuel.
  2. Canonicalise (résout .., les liens symboliques, etc.) pour obtenir une identité stable pour la détection de cycles.
  3. Analyse le fichier importé récursivement en utilisant la même instance de parser et le même ensemble visited.
  4. Fusionne le résultat dans le document actuel en utilisant la sémantique Child-Wins-on-Conflict — si une clé existe déjà dans le fichier importateur, la valeur du fichier importateur est conservée.

Sémantique de fusion​

EntitéComportement à l'importation
Entrées clé-valeurFusionnées ; le fichier actuel gagne en cas de conflit
Définitions @schemaFusionnées dans le registre des schémas
Alias @typeFusionnés dans le registre des types
@import imbriquéRésolu récursivement

Implémentation​

fn process_import(
    import_path: &str,
    current_file: &Path,
    current_doc: &mut AAML,
    visited: &mut HashSet<PathBuf>,
) -> Result<(), AamlError> {
    // 1. Résoudre le chemin relativement au répertoire du fichier actuel
    let raw_path = current_file
        .parent()
        .unwrap_or(Path::new("."))
        .join(import_path);

    // 2. Canonicaliser pour une identité stable
    let canonical = raw_path.canonicalize().map_err(|_| {
        AamlError::FileNotFound { path: raw_path.clone() }
    })?;

    // 3. Détection de cycles — voir la section ci-dessous
    if visited.contains(&canonical) {
        return Err(AamlError::CyclicImport { path: canonical });
    }

    // 4. Analyser le fichier importé récursivement
    visited.insert(canonical.clone());
    let imported_doc = parse_file_recursive(&canonical, visited)?;
    visited.remove(&canonical);

    // 5. Fusionner : l'enfant (actuel) gagne en cas de conflit
    for (key, value) in imported_doc.elements {
        current_doc.elements.entry(key).or_insert(value);
    }
    for (name, schema) in imported_doc.schemas {
        current_doc.schemas.entry(name).or_insert(schema);
    }
    for (alias, ty) in imported_doc.type_registry {
        current_doc.type_registry.entry(alias).or_insert(ty);
    }

    Ok(())
}
remarque

Remarquez l'usage de .entry(...).or_insert(...) — cela garantit que le fichier importateur gagne toujours. Les valeurs du fichier importé ne sont insérées que lorsque la clé n'existe pas encore dans le document actuel.

Protection contre les Dépendances Cycliques​

Anti-Boucle — Cette section est obligatoire à implémenter

Sans détection de cycles, une paire de fichiers qui s'importent mutuellement causera une récursion infinie et un débordement de pile. Cette protection n'est pas optionnelle.

Le problème​

Considérez les deux fichiers suivants :

# a.aam
@import "b.aam"
x = 1
# b.aam
@import "a.aam"
y = 2

Sans protection, analyser a.aam déclenche l'analyse de b.aam, qui déclenche à nouveau l'analyse de a.aam — indéfiniment.

La solution : un ensemble "Fichiers Visités" (Visited Files)​

Maintenez un HashSet<PathBuf> (ou l'équivalent dans votre langage) de chemins canoniques qui sont actuellement en cours de traitement sur la pile d'appels. Avant de traiter un import :

  1. Vérifiez si le chemin canonique est déjà dans l'ensemble.
  2. Si oui — retournez immédiatement une erreur CyclicImport.
  3. Si non — ajoutez-le, faites la récursion, puis supprimez-le après le retour de l'appel récursif.
fn parse_file_recursive(
    path: &Path,
    visited: &mut HashSet<PathBuf>,
) -> Result<AAML, AamlError> {
    let canonical = path.canonicalize()?;

    // ── Protection contre les boucles (Anti-loop guard) ──────────────────────
    if visited.contains(&canonical) {
        return Err(AamlError::CyclicImport {
            path: canonical,
        });
    }
    visited.insert(canonical.clone());
    // ─────────────────────────────────────────────────────────────────────────

    let source = std::fs::read_to_string(&canonical)?;
    let doc = parse_source(&source, &canonical, visited)?;

    // Supprimer APRÈS la récursion pour que les frères puissent continuer à être importés
    visited.remove(&canonical);

    Ok(doc)
}

Pourquoi des chemins canoniques ?​

L'utilisation de chemins bruts n'est pas fiable :

  • ./shared/base.aam, shared/base.aam et ../project/shared/base.aam peuvent tous se référer au même fichier.
  • La canonicalisation les résout tous en un unique chemin absolu, vous donnant une identité stable.

Les imports en diamant sont autorisés​

      main.aam
      /      \
  a.aam     b.aam
      \      /
      base.aam

base.aam est importé à la fois par a.aam et b.aam. Ce n'est pas un cycle — ce sont des chemins de traversée séparés. Le pattern visited-set gère cela correctement car base.aam est retiré de l'ensemble après la fin de la primera branche, et peut donc être entré de nouveau depuis la deuxième branche.

Rapport d'erreurs​

Error: cyclic import detected
  --> a.aam
   |
 1 | @import "b.aam"
   |
   = note: import chain: a.aam → b.aam → a.aam

Reconstituez la chaîne en passant un Vec<PathBuf> à côté du HashSet pour enregistrer l'ordre du parcours.

Mise en Cache​

Analyser le même fichier plusieurs fois lors d'un seul appel load() est inefficace. Maintenez un cache de documents (HashMap<PathBuf, AAML>) séparé de l'ensemble visited :

pub struct ParseContext {
    /// Fichiers actuellement dans la pile d'appels — utilisés pour la détection de cycles.
    visited: HashSet<PathBuf>,

    /// Documents déjà analysés — évite les I/O et les analyses redondantes.
    cache: HashMap<PathBuf, AAML>,
}

Le cache est indexé par chemin canonique. Avant d'analyser un fichier, vérifiez d'abord le cache. L'ensemble visited et le cache ont des objectifs différents et ne doivent pas être confondus.
